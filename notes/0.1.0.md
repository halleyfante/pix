# 0.0.1

## Grid

Define the canvas size in pixels.

```
grid 16 by 16
```

## Colors

Hex colors with `#`. Three formats:

- 3 digits: `#F00`, each digit doubles (`F` becomes `FF`), fully opaque
- 6 digits: `#FF0000`, RGB, fully opaque
- 8 digits: `#FF000080`, RGBA, last two digits are transparency

Named colors are defined in a `color` block and referenced with `color`:

```
color {
    skin: #FFCB96
    eyes: #000000
}

draw x = 3 with color skin
```

## Drawing

`draw` fills every pixel where the condition is true. The condition is checked at every `(x, y)` position on the grid.

```
draw x = 3 and y = 5 with #FF0000
```

`erase` clears pixels where the condition is true. `clear` resets the whole grid.

```
erase x = 0
clear
```

## Expressions

`x` is the horizontal position (0 = left), `y` is the vertical position (0 = top).

### Arithmetic

Works on numbers, produces numbers.

```
x + 5       addition
x - 8       subtraction
2 * x       multiplication
x / 2       integer division (rounds toward zero)
x ^ 2       exponentiation
```

### Comparison

Takes two numbers, produces true or false.

```
x = 3       equal
x < 5       less than
x > 0       greater than
x <= 5      less than or equal
x >= 0      greater than or equal
```

### Logic

Combines true/false values.

```
x = 3 and y = 5       both must be true
x = 0 or y = 0        at least one must be true
not (x = 0)            inverts true/false
```

### Precedence

From lowest to highest: `or`, `and`, `not`, comparisons, `+` `-`, `*` `/`, `^`. Parentheses override.

`x + 1 < 5 and y > 0` is read as `((x + 1) < 5) and (y > 0)`.

`2 ^ 3 ^ 4` is read as `2 ^ (3 ^ 4)`. Exponentiation groups right to left.

## Shapes

Shapes are shorthands. Each one is a mathematical condition on `x` and `y`.

### Pixel

A single point. True only where `x` and `y` both match exactly.

```
pixel (3, 5) with #FF0000
```

The math: `x = 3 and y = 5`.

### Line

All points between two endpoints, including the endpoints themselves.

```
line (0, 0) to (8, 8) with #FF0000
```

For a horizontal line like `line (2, 5) to (7, 5)`: `y` must be 5, and `x` must be between 2 and 7 (inclusive). Same idea for vertical.

For a diagonal line, the math checks if the point is exactly on the line using the cross product. Take two points A and B. For any point P, compute:

```
(B.x - A.x) * (P.y - A.y) - (B.y - A.y) * (P.x - A.x)
```

If this equals zero, P is on the line. The result is also bounded by the x and y ranges of A and B so the line doesn't extend forever.

### Rectangle

The interior of a rectangle, not including the border pixels.

```
rectangle (1, 1) to (5, 5) with #0000FF
```

The math: `x > 1 and x < 5 and y > 1 and y < 5`. Pixels at 1 and 5 are not drawn, only 2, 3, 4.

Point order doesn't matter. `rectangle (5, 5) to (1, 1)` gives the same result.

### Triangle

The interior of a triangle, not including the edges.

```
triangle (0, 0) to (10, 0) to (5, 10) with #FF0000
```

A triangle has three edges. Each edge is a line that splits the grid into two halves. A pixel is inside the triangle if it's on the correct side of all three edges.

To determine which side a point is on, we use the same cross product as for lines. For each edge, the cross product gives a positive or negative number: positive means one side, negative means the other. We check the third vertex (the one not on the edge) to know which sign means "inside."

### Circle

The interior of a circle, not including the exact border.

```
circle (8, 8) radius 4 with #00FF00
```

A pixel is inside if its distance to the center is less than the radius. Instead of computing the actual distance (which needs a square root), we compare squared values:

```
(x - 8)^2 + (y - 8)^2 < 4^2
```

`(x - 8)^2` is how far the pixel is horizontally from the center, squared. Same for `(y - 8)^2` vertically. Add them together and you get the squared distance. If that's less than the radius squared (16), the pixel is inside.

## Copy

Copy a rectangular region of the grid to another position. The original remains.

```
copy (0, 0) to (7, 7) at (8, 0)
```

## Move

Move a rectangular region of the grid to another position. The source region is erased.

```
move (0, 0) to (7, 7) at (8, 0)
```

## Layer

Named block that groups drawing statements onto an isolated grid. `clear` and `erase` inside a layer only affect that layer. Layers are composited in order, later layers are drawn on top.

```
layer background {
    rectangle (0, 0) to (15, 15) with #87CEEB
}

layer character {
    circle (8, 8) radius 3 with #e84a00
}
```

An `export` inside a layer block exports only that layer. An `export` outside any layer exports all layers composited. In SVG, each layer becomes a `<g>` element with the layer name as `id`.

## Mirror

Block that draws its contents and also their reflection across a two-point axis.

```
mirror (0, 8) to (15, 8) {
    circle (4, 4) radius 3 with #e84a00
}
```

The two points define the axis line. Any two points on the grid are valid.

## Frame

Captures the current grid state as an animation keyframe. Each frame requires a delay in milliseconds.

```
circle (4, 4) radius 1 with #e84a00
frame 200

circle (4, 4) radius 2 with #e84a00
frame 100

circle (4, 4) radius 3 with #e84a00
frame 500

export "grow" in gif
```

The grid accumulates between frames. Use `clear` to reset explicitly. If `frame` statements are present but no animated export is used, the compiler emits a warning.

## Export

Writes the grid to an image file. Four formats are supported:

- `png`: lossless raster image with transparency.
- `svg`: vector format. Each pixel becomes a `<rect>` element. Scales without loss of quality.
- `webp`: compact raster format for the web.
- `gif`: animated format. Collects all `frame` keyframes into an animation.

`scale` multiplies each pixel. A 16x16 grid with `scale 4` produces a 64x64 image. For SVG, scale controls the size of each `<rect>` element.

```
export "art" in png
export "art" in svg
export "art" in webp
export "animation" in gif
export "art-big" in png scale 4
```

Transparent pixels (not drawn) become fully transparent in PNG and WebP. In SVG, no `<rect>` is emitted for empty pixels.

## Comments

Everything after `//` until the end of the line is ignored.

```
// this is a comment
draw x = 0 with #FF0000  // inline comment
```

## Tokens

The full set of tokens the language recognizes:

- Numbers: `0`, `16`, `255`
- Hex colors: `#FF0000`
- Strings: `"filename"`
- Identifiers: custom names used in color blocks
- Operators: `+`, `-`, `*`, `/`, `^`, `=`, `<`, `>`, `<=`, `>=`
- Keywords: `and`, `or`, `not`, `with`, `in`, `by`, `to`, `radius`, `scale`, `color`, `x`, `y`, `at`
- Statements: `draw`, `erase`, `clear`, `grid`, `export`, `frame`, `copy`, `move`
- Blocks: `layer`, `mirror`
- Shapes: `pixel`, `line`, `rectangle`, `triangle`, `circle`
- Formats: `png`, `svg`, `webp`, `gif`
- Delimiters: `(`, `)`, `{`, `}`, `:`, `,`
- Comments: `//` to end of line

## Language Server

LSP server over stdio. Handles `initialize`, `textDocument/completion`, and `shutdown`. Completion is context-aware based on the token before the cursor.

## Extension

VS Code extension with TextMate syntax highlighting and language server integration. The language server binary is bundled in the extension package.

## Playground

Web editor with Monaco and a canvas preview. The server receives source code, runs the compiler, and returns the rendered PNG. Autocomplete is proxied through the language server. Deployed as a Docker container.
